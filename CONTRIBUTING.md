# Contributing to PRD Generator

First off, thank you for considering contributing to PRD Generator! It's people like you that make this tool better for everyone.

## How Can I Contribute?

### Reporting Bugs

Before creating bug reports, please check the existing issues as you might find out that you don't need to create one. When you are creating a bug report, please include as many details as possible:

- **Use a clear and descriptive title**
- **Describe the exact steps to reproduce the problem**
- **Provide specific examples to demonstrate the steps**
- **Describe the behavior you observed after following the steps**

### Suggesting Enhancements

Enhancement suggestions are tracked as GitHub issues. When creating an enhancement suggestion, please include:

- **Use a clear and descriptive title**
- **Provide a step-by-step description of the suggested enhancement**
- **Provide specific examples to demonstrate the steps**
- **Describe the current behavior and explain the behavior you expected**

### Adding New Templates

We welcome new PRD templates for different industries or use cases:

1. Fork the repository
2. Create a new template file in `templates/` directory
3. Follow the naming convention: `[industry]-prd-template.md`
4. Submit a pull request

### Adding New Examples

Examples help users understand how to use the skill:

1. Fork the repository
2. Create a new example file in `examples/` directory
3. Follow the naming convention: `sample-[product-type].md`
4. Submit a pull request

### Pull Requests

- Fill in the required template
- Do not include issue numbers in the PR title
- Include screenshots and animated GIFs in your pull request whenever possible
- Follow the markdown style guide
- End all files with a newline

## Style Guides

### Git Commit Messages

- Use the present tense ("Add feature" not "Added feature")
- Use the imperative mood ("Move cursor to..." not "Moves cursor to...")
- Limit the first line to 72 characters or less
- Reference issues and pull requests liberally after the first line

### Markdown Style Guide

- Use [GitHub Flavored Markdown](https://guides.github.com/features/mastering-markdown/)
- Use fenced code blocks instead of indented code blocks
